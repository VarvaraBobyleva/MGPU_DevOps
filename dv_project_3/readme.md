# Оркестрация контейнеризированных приложений в Kubernetes

## 🎯 Цель работы
Получение практических навыков миграции архитектуры Docker Compose в среду Kubernetes (Minikube). Настройка конфигураций, обеспечение персистентности данных и управление жизненным циклом сервисов.

---

## 🏗 Архитектура решения

Согласно требованиям, система разделена на логические блоки, представленные в таблице ниже:


| Блок | Компонент | Краткое пояснение |
| :--- | :--- | :--- |
| **Configs** | Secret / ConfigMap | Хранилище паролей (base64) и несекретных настроек БД (хост, порты). |
| **Database** | PostgreSQL / PVC | База данных. **PVC** обеспечивает сохранность данных при перезапуске подов. |
| **Analytics** | Streamlit App | Аналитическая среда. Использует **InitContainer** для ожидания БД и **Probes** для проверки статуса. |
| **Data** | Loader Job | Однократный процесс (**K8s Job**), наполняющий БД данными из CSV-файла. |
| **User** | Analyst | Доступ к Dashboard через сервис типа **NodePort** (внешний доступ). |

## 🛠 Реализация ключевых критериев

### 1. Манифесты и Сетевое взаимодействие
- **Deployment**: Описаны для `db` (PostgreSQL) и `crypto-app` (Streamlit).
- **Service**: 
  - `db-service` (ClusterIP) — внутренняя связь.
  - `app-service` (NodePort) — доступ к дашборду извне.
- Все YAML-файлы валидны и применены одной командой `kubectl apply -f k8s/`.

### 2. Конфигурация и Безопасность
Полный отказ от хардкода переменных окружения:
- **Secret (`crypto-secrets`)**: Хранит `DB_PASSWORD` в кодировке base64.
- **ConfigMap (`crypto-config`)**: Хранит `DB_HOST`, `DB_NAME`, `DB_USER`.
- В подах данные подключаются через `valueFrom` (secretKeyRef/configMapKeyRef).

### 3. Хранение данных (Storage)
Для базы данных настроен **PersistentVolumeClaim (PVC)** объемом 1Gi. 
> **Тест на персистентность:** При удалении пода базы (`kubectl delete pod <db-pod>`) K8s автоматически создал новый. Благодаря PVC данные в таблице `crypto_stats` сохранились, и дашборд продолжил отображать графики без повторной загрузки.

### 4. Жизненный цикл и Probes
- **InitContainer**: Контейнер `wait-for-db` на базе `busybox` проверяет доступность порта 5432 перед запуском основного приложения.
- **Liveness/Readiness Probes**: Настроены для приложения (порт 8501) для автоматического перезапуска при зависании.

### 5. Специфика варианта (Resource Quotas & Job)
- **Job**: Реализован `loader-job.yaml` для разовой миграции данных из датасета в БД.
- **Ресурсы**: Для пода приложения установлены лимиты:
  - `cpu`: 500m (limit) / 100m (request)
  - `memory`: 256Mi (limit) / 128Mi (request)
  *Это предотвращает "поедание" ресурсов всего кластера тяжелыми аналитическими запросами.*

## 📊 Выводы по анализу данных

В ходе выполнения работы были реализованы механизмы обработки и визуализации временных рядов (Time-series) криптовалютных активов. Основные результаты:

### 1. Обработка данных (Backend)
* **Алгоритм выборки**: Реализован механизм **случайных временных окон** (Random Time Window) из БД PostgreSQL. Это позволяет анализировать динамику актива в разные периоды, не перегружая оперативную память контейнера (соблюден лимит **256Mi**).
* **Оптимизация**: Использована библиотека **SQLAlchemy** для выполнения агрегирующих запросов на стороне базы данных, что снижает нагрузку на клиентское приложение.

### 2. Визуализация (Frontend)
* **Динамика цены (Line Chart)**: Интерактивный график позволяет отслеживать волатильность и тренды актива (Asset ID: 1). Наглядно зафиксирован рост котировок в выбранном интервале (с 46.4k до 47.1k).
* **Анализ объемов (Area Chart)**: Выявлены импульсные всплески торговой активности, коррелирующие с моментами резкого изменения цены.

### 3. Итоговый результат
* Создана **масштабируемая аналитическая платформа** в среде Kubernetes.
* Подтверждена работоспособность всей цепочки: **Loader (Job)** успешно наполняет базу, **DB (PVC)** сохраняет данные, а **App (Deployment)** корректно их визуализирует.
